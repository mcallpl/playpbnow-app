CI/CD Pipeline — The Meta-Bug

Fixed & Verified Live
Critical.github/workflows/deploy.yml:210 (Deploy web files)

Every push to main re-breaks production. The "Deploy web files" step runs rsync --delete dist/ → /var/www/html/PlayPBNow/. Because dist/ contains no api/ folder, --delete wipes the entire live api/ directory — db_config.php, vendor/, and all PHP endpoints. The subsequent "Backup critical files" step runs after the nuke, so it captures nothing; the API redeploy then excludes db_config.php/vendor (correctly, since they're not in the repo) and the restore step has nothing to restore. Net result on every push: API returns 500 for everything. This is what took prod down again between my last two turns — my own end-of-session git push triggered it. Fix: add --exclude=api to the web rsync so --delete never touches the backend, plus a hard guard that restores db_config/vendor from the latest backup if they ever go missing.

Criticaldeploy.yml build step + app.json

The web build never set the Expo baseUrl, so the exported app.html referenced its JS bundle at /_expo/... (domain root) instead of /PlayPBNow/_expo/... — 404, blank screen — on every CI deploy. Fixed by committing experiments.baseUrl: "/PlayPBNow" into app.json so every build (CI and local) produces correct paths. Caveat (documented follow-up): project memory notes baseUrl breaks native iOS/EAS builds — before the next iOS build, baseUrl must be removed or moved into an app.config.js that only sets it for web. Noted so it isn't forgotten.

Hightwo-repo architecture

The deployed API is a separate GitHub repo (github.com/mcallpl/playpbnow-api), which the pipeline clones fresh on each deploy — NOT this app repo's playpbnow-api/ folder. Verified: the separate repo matches the live server byte-for-byte (0 differing files), while this repo's copy has drifted (20 files differ, 23 exist only here including undeployed auth-hardening WIP). Consequence: all API/PHP bug fixes must be made in the separate repo to actually reach production; edits to this repo's playpbnow-api/ are dead. This also explains why the earlier "fix agents" appeared to fix things that never changed in production — they edited the wrong repo.

Fixed & Verified

Production Incident — Resolved

Fixed & Verified Live
Root Cause/var/www/html/PlayPBNow/api/db_config.php + api/vendor/

The entire backend API has been completely down since approximately June 25-26, 2026 — every single endpoint was returning HTTP 500. The file every PHP endpoint requires to connect to the database (db_config.php) and the Composer vendor/ directory (Twilio/Stripe SDKs) were both simply missing from the production server — most likely wiped by a deploy that rsynced the API directory without the backup/restore safety step actually preserving them (a later commit, "Properly backup and restore critical API files during deployment," appears to be an attempted fix for this exact class of bug, but landed too late to restore what was already gone). This is almost certainly the true explanation for "the app is ridden with bugs" — matches, invites, login, everything that touches the API would have been failing for about a week. Restored both from the last known-good server backup (2026-06-25 00:51, checksum-verified identical to the local repo copy) and confirmed live: get_courts.php and others now return real 200 JSON responses.

Contributing Bugweb build — missing baseUrl at export time

Separately, the last web deploy exported the Expo web bundle without setting the baseUrl experiment first, so the built app.html referenced its JS bundle at an absolute domain-root path (/_expo/...) instead of the actual deployed subdirectory (/PlayPBNow/_expo/...) — since peoplestar.com hosts multiple apps at the same domain, this 404'd against a different app's territory, leaving every visitor with a blank white screen. Root cause traced to .github/workflows/deploy.yml's "Prepare SPA structure" step: it runs fix-expo-paths.js/add-base-tag.js AFTER renaming the real SPA export to app.html and overwriting index.html with the landing page — so those path-fixing scripts have only ever operated on the landing page, never on the actual app bundle. Rebuilt locally with baseUrl: "/PlayPBNow" set correctly, redeployed, verified live.

Contributing Bugpublic/landing.html

The marketing landing page itself (recently touched by the OG-image commits) had 16 absolute root-path links/assets (/app.html, /privacy.html, /manifest.json, app icons, etc.) that all 404'd for the same subdirectory-vs-root reason — so even before the JS bundle broke, visitors clicking "Launch App," "Privacy," "Terms," or "Support" from the marketing site would have hit dead links. Fixed all 16 references to the correct /PlayPBNow/ prefix and redeployed.

Fixedplaypbnow-api/purge_data.php

Added real authentication (Bearer session token + admin-flag check against the database) and restricted the method to POST — closes the unauthenticated GET-triggerable data-wipe hole reported in API Backend Security below. Verified: unauthenticated GET now returns 405, unauthenticated POST now returns 401.

Systemic Bug — Fixed29 files: hardcoded const API_URL = '/api'

Every single API call the web app makes was hitting the wrong URL. 29 separate files (app screens, hooks, components) each independently hardcoded API_URL = '/api' — a domain-root-relative path. Under the actual deployment (a subdirectory, /PlayPBNow/), this resolves to the wrong location entirely (confirmed live: registration POSTed to peoplestar.com/api/email_login.php instead of peoplestar.com/PlayPBNow/api/email_login.php, 404). This is very likely the single biggest contributor to "the app is ridden with bugs" — it would have broken every feature that talks to the server. Fixed all 29 files to use the absolute production URL and redeployed; verified registration/login/groups/players/rankings all work end-to-end now.

Code Drift Discoveredplaypbnow-api/email_login.php (live vs. local repo)

The live server's copy of this file differs from the local repo copy — someone edited it directly on the server at some point, and that edit was never synced back. The live-only version required a phone number for registration ("Phone number is required"), directly contradicting the documented product decision ("email-based registration, phone optional") and blocking every phone-less signup. Made a minimal, dependency-free live fix (removed the phone-required block) to restore the documented behavior. Separately: the local repo's version of this file is a much more advanced, hardened rewrite (rate limiting, account lockout, CSRF tokens, restricted CORS) from the June 24 "Auth Hardening" work — but it depends on a security_helpers.php file and database columns (user_sessions.device_name, .is_active) that do not exist on the production server. That entire hardening effort was built but never actually deployed. Deploying it safely would require creating the missing helper file and running a schema migration first — flagged here as a recommended follow-up, not done automatically.

Awaiting your decisionproduction DB: users.phone column

Even with the code fixed, phone-less registration still fails at the database level: users.phone is VARCHAR(20) NOT NULL UNIQUE. I asked whether to run ALTER TABLE users MODIFY phone VARCHAR(20) NULL; (safe, additive, no data loss — MySQL allows multiple NULLs in a UNIQUE index) and didn't get a response in time, so I left it as-is and used a test phone number to keep testing. This is the last piece needed to make "phone optional" actually true end-to-end — let me know if you want it applied.

Noteserver disk usage

The production server's disk is at 90% (52G/58G) — a large part is /var/backups/PlayPBNow-backup-* snapshots accumulated from every past deploy attempt (22+ found). Worth a cleanup pass separate from this audit; not blocking, but if it fills up it will take deploys and backups down with it.

API Backend Security

Critical — Multiple Unauthenticated Endpoints
Criticalplaypbnow-api/purge_data.php

Confirmed live on production (HTTP 500 response received, endpoint exists — did not trigger the actual purge). A plain GET request with ?confirm=YES TRUNCATEs matches, sessions, groups, player_group_memberships, feature_access, payment_transactions, subscriptions, user_sessions, verification_codes and more. No auth, no admin check, wildcard CORS, and it's a GET — trivially triggerable via a link, an <img> tag, or a crawler bot. This is the single most dangerous finding in this entire audit.

Criticalactivate_subscription.php

Grants full Pro subscription + unlocks all feature_access flags for any user_id in the POST body, with zero authentication or webhook signature check — despite its own comment claiming it's only "called by RevenueCat webhook." Anyone can grant themselves Pro for free, or revoke someone else's. The real Stripe/RevenueCat webhooks (stripe_webhook.php, revenuecat_webhook.php) DO verify signatures correctly — this is a leftover unlocked backdoor.

Criticaladmin_api.php:16-27, broadcast_api.php:30-41

Admin check trusts a client-supplied user_id with no session verification — anyone who knows or guesses an admin's numeric user_id gets full admin dashboard and broadcast access.

Criticalwhole architecture

The June 24 auth/CORS/transaction fixes were built into a parallel API architecture (routes.php → Middleware.php → controllers/) that the app never calls. Every real screen fetches the legacy flat .php files directly (confirmed in JoinMatchModal.tsx, hooks/useGameSession.ts, etc.), which have almost none of the claimed protections. The working AuthMiddleware, CORS allow-list, and transactional SMS helper all exist in the codebase — just wired to nothing.

HighStill Broken73 of ~79 PHP endpoint files

CORS wildcard confirmed at greater scale than the original audit found — 73 files still use Access-Control-Allow-Origin: *. Only 6 files (forgot_password.php, manage_sessions.php, logout.php, change_password.php, email_login.php, validate_session.php) correctly restrict to peoplestar.com.

Highdelete_player.php, delete_group.php:26-29

delete_player.php deletes any player by ID with zero auth. delete_group.php's ownership check only runs "if ($user_id)" is present — simply omitting it from the request skips the check and deletes any group.

Highsend_verification_code.php

No rate limiting at all — anyone can trigger unlimited verification SMS to any phone number at Twilio's expense, no auth required. (The one endpoint that does rate-limit correctly, forgot_password.php, shows the team knows the pattern — it just wasn't applied here.)

Mediummigrations/, apply_indexes.php

Index-creation scripts exist and are well-designed but are manual-only — deploy-api.sh never runs them, so there's no evidence they were ever applied to the production database.

Mediumdebug_players.php, check_schema.php, diag.php, fix.php, etc.

Debug/diagnostic scripts are deployed verbatim to production with no auth and wildcard CORS, several likely exposing internal schema info.

Confirmed Fixed

join_match.php's batch_id→id bug is a real, correct fix (though the endpoint it lives in is otherwise unauthenticated and mostly unused — see Matches section).

Highcollab_sync_scores.php:53-94

Race condition still live in production. The real scoring endpoint does an unlocked read-then-write with no transaction — two devices editing the same game within the same second can silently overwrite each other's score. The correct transactional pattern (SELECT...FOR UPDATE + commit) exists in update_match.php, but that file isn't on the live-scoring path.

Highdelete_match.php + hooks/useLeaderboardLogic.ts:285-296

Delete-match is functionally broken for everyone. The server requires user_id in the request body to authorize a delete, but the only frontend caller never sends it — every delete request fails regardless of who owns the match. Separately, for any match where created_by_user is NULL (common on older rows), the ownership check is skipped entirely and anyone can delete it. Sibling endpoint update_match.php has no identity check at all — authorizes purely by guessable group name.

Mediumhooks/useCollaborativeScoring.ts:5,300 vs hooks/useScoreSync.ts

The "10s interval with exponential backoff" commit is real but landed in useScoreSync.ts, which is never imported anywhere. The hook actually wired into the live game screen polls every 2 seconds, fixed, forever, with no backoff on errors.

Mediumplaypbnow-api/test_match_fixes.php

The verification script used to certify these fixes only greps source text for keywords (e.g. "does the string begin_transaction appear in this file?") — it never issues an HTTP request or exercises the actual routes. This is how "fixed" got reported for code that's unreachable in production.

Lowcollab_create_session.php

No auth check at all on session creation (accepts any creator_user_id from the body), no rate limiting. Share codes are 6 characters from a 24-letter alphabet (~191M combos) with no brute-force protection on join/get-scores endpoints.

Invites / SMS

Critical Issues
Criticalevery live invites endpoint

Root cause of nearly every finding below: no authentication anywhere in this feature. invite_api.php, pool_players_api.php, invite_respond.php, invite_chat_send.php, invite_chat_poll.php, sms_credits_api.php, and freestyle_sms_api.php all trust a client-supplied user_id/player_id in the JSON body with zero session/token verification — even though a working auth mechanism (Middleware.php's verifyToken() against user_sessions) already exists elsewhere in the codebase and is simply never wired in here.

Highinvite_respond.php

IDOR on invite responses. The only thing gating who can confirm/decline as a given player is player_id — a small, sequential, publicly-guessable integer embedded in plaintext in the SMS link. Anyone who receives one invite link can edit the URL to respond on behalf of any other invited player. Also: response changes trigger free organizer-notification and waitlist-promotion SMS sends with zero credit deduction and no rate limit — an unauthenticated cost-abuse vector against the account owner's Twilio bill.

Highinvite_api.php:130-187 (send action)

Credit-check race condition. Balance is checked once before a loop, with no transaction, no row lock, and no floor guard on the decrement. Two concurrent send requests can both pass the balance check and both deduct, allowing more SMS to be sent than paid for. A correct, transactional version (sendSMSWithTransaction() in db_config.php) already exists in the codebase but is only wired to a dead-code controller, never to this live endpoint.

Highfreestyle_sms_api.php

Admin check is spoofable (trusts client-supplied user_id). If exploited: dumps raw unmasked phone+email for the entire pool, and allows arbitrary free-text SMS to any pool player with no credit deduction and no rate limit.

Highinvite_chat_send.php / invite_chat_poll.php

No auth, no ownership check, sequential guessable invite_id. Anyone can post messages into any invite's chat under any name, or read any invite's full chat history. CORS wildcard confirmed still present (previously flagged, still broken).

Mediumdb_config.php:17-25, 337-345

Real production Twilio SID/Auth Token are hardcoded as a silent fallback if the vault file is missing, rather than failing loudly — a misconfigured deploy could silently run on baked-in prod secrets.

Confirmed Good

Match codes use a real CSPRNG (random_int, 6 chars/32-letter alphabet, ~1B combos) with DB uniqueness checks. Failed Twilio sends correctly do NOT deduct a credit in the live send loop. No evidence any real SMS was sent or credits deducted during this review.

Mediumarchitecture-wide

The app runs on a legacy set of flat PHP scripts while a better-engineered, authenticated MVC stack (controllers/, routes.php) sits unused alongside it. This is why fixes keep landing in the wrong place — recommend finishing the cutover or deleting the dead stack to stop this pattern from repeating.

Stripe / SMS Credit Purchases

Critical Issues
Criticalplaypbnow-api/stripe_webhook.php:12-33

Webhook signature check fails open. Verification only runs "if a Stripe-Signature header is present." A request with NO signature header skips verification entirely and is processed as trusted. Since credit-purchase handling reads credits and user_id straight from attacker-controlled JSON metadata, anyone can POST directly to this endpoint with a plain curl command and grant themselves unlimited free SMS credits — no real Stripe transaction required. This directly contradicts the June 24 "zero vulnerabilities" claim.

Highplaypbnow-api/sms_credits_api.php (whole file)

No authentication on the credits endpoint at all. Unlike the rest of the API (which validates a Bearer token via Middleware.php), this file reads user_id straight from the client-supplied JSON body. Any caller can read another user's SMS credit balance and full purchase history (IDOR), or create a checkout session that credits an arbitrary account. Also sets Access-Control-Allow-Origin: * and has no rate limiting.

Highdb_config.php vs stripe_webhook.php vs tests/DatabaseFixtures.php

Schema drift may make purchased credits unspendable. The purchase path writes to a credits column (sms_credits table); the actual SMS-send deduction path (sendSMSWithTransaction(), used by InviteController) reads a balance column instead. If production's real schema matches the deduction path, every invite send would incorrectly report "Insufficient SMS credits" regardless of purchase history. Needs verification against the live DB schema and an end-to-end purchase→spend test.

Mediumstripe_webhook.php (credit + subscription branches)

No transaction/row-lock around the check-then-insert duplicate guard — concurrent webhook retries (which Stripe does on any non-2xx) could double-credit. The subscription-renewal branch has no idempotency guard at all; replaying a captured valid signature extends a subscription for free.

Highsms_credits_api.php:24-25

Checkout session uses relative success_url/cancel_url ("/?credits=success") — Stripe's API requires fully-qualified URLs. If unconfirmed against server logs, this may mean checkout-session creation fails for every purchase attempt in production.

Confirmed Good

No live Stripe secret keys or webhook secrets are committed to source control — they load correctly from an untracked vault file. (Note: Twilio/Google Maps/Gemini keys ARE hardcoded with fallback values directly in the tracked db_config.php — a separate, lower-severity secret-hygiene gap.)

Live Browser Test (Production)

Critical Crash Found
FIXEDapp/setup/steps/PlayerInputStep.tsx

Player-add crash — root cause was deeper than first thought, now fixed. The real cause: this file rendered <Text> (in the search-results dropdown) but never imported Text from react-native — so on web the bare identifier resolved to the browser's global window.Text DOM constructor, and React threw Failed to construct 'Text' whenever the dropdown rendered (i.e. whenever a typed name matched pool players — which is why it seemed intermittent). Fixed by adding the missing Text import. Swept the whole codebase (app/ + components/) for the same missing-import pattern — this was the only offender. Also replaced the duplicate-name confirmation (which used a web-broken multi-button Alert.alert) with a proper cross-platform modal. Verified the build compiles; live re-test pending deploy.

FIXEDreact-native-web Alert.alert (app-wide, 159 call sites)

Discovered while fixing the above: every confirmation and error dialog is silently dead on the web app. react-native-web 0.21.2 implements Alert.alert as a literal no-op (static alert() {}). That means Log Out, Delete Account, Delete Group/Player/Beacon, Merge Players, Send Broadcast, score-update confirmations, and all error messages do nothing on web — a large slice of "the app is broken." Fixed with a drop-in cross-platform Alert shim (utils/crossAlert.ts): unchanged native behavior, and on web it maps to real window.confirm/window.alert dialogs so confirmations and errors actually work. Swapped the import in all 27 affected files (no call-site churn). Build verified.

Highapp.json experiments.baseUrl (build-time), app/login.tsx registration UX

Registration itself works end-to-end after today's fixes (verified: create account → auto-login → land on Groups). Player search-as-you-type, group creation, location picker, player pool browsing, invites tab, and the global rankings/leaderboard all loaded and worked correctly with real data.
